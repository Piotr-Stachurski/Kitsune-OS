Genesis

Built From Operational Pain

After sending 200+ applications, manual tracking became the bottleneck. I started with Google Sheets, escalated through scripted Gemini prompts, then attempted a custom Gemini GEM — which proved unreliable for production use (hallucinations, broken duplicate detection, lost session context).

I iterated through three off-the-shelf AI configurations before identifying that custom architecture was the lower-effort path to reliability. The pattern was familiar: an operational pain point I'd solved for clients in enterprise — applied to my own pipeline, at zero infrastructure cost, during an active job search.

This is not a side project. It's a production system serving a live use case — dogfooded against the exact problem it was designed to solve.

FAILURE MODE 01
Data Fragmentation
No Single Source of Truth. Pipeline visibility collapses under unstructured, multi-source data streams. Decision velocity drops to near zero.
FAILURE MODE 02
Cognitive Overload
Unstructured input saturated with corporate noise ("BS Bingo"). High time-cost per analytical decision. GIGO risk in every manual evaluation cycle.
FAILURE MODE 03
Infrastructure Reliability
LLM API throttling, cloud migration race conditions, and burst throughput limits cause silent data loss and broken analytical pipelines.

Engineering Decisions

Why It Was Built This Way

Why refactor Stage 2 from 3 nodes to a single unified call?
V1 used 3 parallel nodes for fault isolation — each failing independently without losing the others. After measuring production behavior, Gemini Flash demonstrated stable JSON adherence under a single unified prompt at temperature 0.6. Refactor result: 80% cost reduction (20 gr → 4 gr PLN per full cycle / $0.06 → $0.01 ). Single-point-of-failure risk accepted, mitigated by Kitsune-Shield 4-layer fallback JSON parser. Engineering decisions follow measurement, not assumptions.
Why Exponential Backoff over fixed-delay retry?
GAS has a hard 6-minute execution limit. Drive and Sheets APIs have burst throttling. Fixed-delay retries waste quota by hitting the same throttled window. Exponential backoff (4s → 8s → 16s → 32s → 64s) finds the API's recovery window. Fast-track switch (zero delay) for infrastructure errors (503, 429, quota) — different error class, different recovery strategy.
Why the Session ID pattern in the frontend?
GAS callbacks are asynchronous and non-cancellable. A mid-analysis reset renders two result sets simultaneously — ghost data in the DOM. Timestamped Session IDs invalidate callbacks from superseded sessions before they render, eliminating this entire class of UI bug. Also applied in Stage 2 to guard against reset during deep-dive processing.
Why sanitizePayload on BOTH frontend AND backend?
Defense-in-depth. This is not redundancy — it's independent layers. If the frontend sanitizer is bypassed (direct API call, future mobile client, JS error), the backend layer prevents Base64 binary data from reaching the LLM context window regardless. Each layer is independently effective. No single point of failure in the data hygiene pipeline.
Why Google Sheets as primary SSoT vs. a database?
Zero deployment complexity. Sheets provides native filtering, formula-driven KPIs, data validation, shareable access, and live dashboard without schema management or hosting costs. For target volume (hundreds of records), it outperforms a dedicated DB on every practical operational metric. Over-engineering is a failure mode too.
Why the Kill-Switch (Autopilot ON/OFF)?
In agentic systems, runaway automation is a real failure mode — not a theoretical one. A hardware-style binary toggle persisted in PropertiesService, visible in the UI at all times, ensures the architect retains absolute control. The human is the SSoT Architect. The AI is the workforce. Governance is a first-class engineering requirement, not an afterthought.
